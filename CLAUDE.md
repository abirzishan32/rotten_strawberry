@AGENTS.md

## What this is

A Letterboxd-inspired movie review/diary app built on Expo Router, styled with NativeWind (Tailwind for React Native). Movie data is live from TMDB; user data (auth, profiles, reviews/diary, favorites, watchlist) is backed by **Supabase** (Postgres + Auth + RLS). Only `theme` preference and `recent searches` persist locally (Zustand + AsyncStorage). Browsing is open to guests; personal actions prompt sign-in.

## Commands

- `npm start` — start the Expo dev server (Metro). Press `i`/`a`/`w` in the terminal to open iOS/Android/web.
- `npm run ios` / `npm run android` / `npm run web` — start the dev server targeting a specific platform directly.
- `npm run lint` — run ESLint via `expo lint` (flat config in `eslint.config.js`, extends `eslint-config-expo`).
- `npx tsc --noEmit` — typecheck. Route param types depend on `.expo/types/router.d.ts`, which is regenerated by `expo start`/`expo export` whenever `app/` changes — run the dev server at least once after adding/removing routes before trusting typed-route errors.
- `npm run reset-project` — leftover from the original template; do not run, it would wipe this app's `app/` directory.
- No test runner is configured in this project.

## Environment / secrets

TMDB auth uses a **v4 Read Access Token** (bearer), stored in `.env` as `TMDB_API_KEY` (gitignored). `app.config.ts` reads `process.env.TMDB_API_KEY` at config-eval time (Expo CLI auto-loads `.env` into `process.env`, no `EXPO_PUBLIC_` prefix needed since it's read server-side) and injects it into `extra.tmdbApiKey`. `services/tmdb/client.ts` pulls it back out via `Constants.expoConfig.extra.tmdbApiKey` and sends it as `Authorization: Bearer <token>` — this keeps the raw key out of app.json/JS source, though it's still embedded in the compiled app config (acceptable for a client-only TMDB read token).

Supabase uses the same `.env` → `app.config.ts` `extra` → `Constants.expoConfig.extra` pattern: `SUPABASE_URL` and `SUPABASE_ANON_KEY` (anon key only — public-safe because RLS guards the data). **Never add the `service_role` key** to `.env` or the app; it bypasses RLS. After editing `.env`, restart the dev server so `app.config.ts` re-reads it.

## Architecture

- **Routing (`app/`)**: `app/_layout.tsx` is the root `Stack` wrapping the app in `GestureHandlerRootView` → `SafeAreaProvider` → `QueryClientProvider` (client from `services/query-client.ts`) → navigation `ThemeProvider` → `ErrorBoundary`. `(tabs)` is the anchor group with 4 tabs (Home, Search, Activity, Profile) defined in `app/(tabs)/_layout.tsx`. `movie/[id]`, `browse/[category]`, `browse/genre/[id]` are pushed outside the tabs group (so the tab bar hides automatically), and `add-log` / `settings` / `auth` are `presentation: 'modal'` screens registered directly in the root `Stack`.
- **Bottom sheets use `BottomSheet`, not `BottomSheetModal`**: `components/common/bottom-sheet.tsx` wraps `@gorhom/bottom-sheet`'s plain, fully-controlled `BottomSheet` (open/closed via an `index` prop) rather than `BottomSheetModal`'s portal + imperative present/dismiss stack, which was dropped after it got stuck on a close/reopen cycle. There is no `BottomSheetModalProvider` in the tree — don't add one or reach for `BottomSheetModal` for new sheets.
- **Logging a movie**: there is no "Add Log" tab. The `add-log` modal (`app/add-log.tsx`) is reached from the "Log" FAB on `movie/[id]` (search → movie → log). It's registered as a modal in the root `Stack`.
- **TMDB layer**: `services/tmdb/client.ts` (axios instance) → `services/tmdb/movies.ts` (one function per endpoint: lists, details, credits, videos, similar, recommendations, search, discover, genres) → `hooks/queries/*` (TanStack Query wrappers; `use-movie-list.ts` deliberately has *two* query keys — `movieList` for single-page home carousels and `movieListInfinite` for paginated browse grids — sharing one key would corrupt the cache since `useQuery` and `useInfiniteQuery` store different shapes). `constants/query-keys.ts` is the single source of truth for query keys.
- **Supabase backend (`services/supabase/`)**: auth + all user-generated data (profiles, reviews/diary, favorites, watchlist) live in Supabase. `client.ts` builds one `createClient<Database>` with the AsyncStorage session adapter + `detectSessionInUrl: false` and an `AppState` auto-refresh listener; per-domain modules (`auth`, `profiles`, `reviews`, `favorites`, `watchlist`) wrap the queries and map snake_case rows ↔ the app's camelCase types via `types.ts`. `hooks/queries/use-{reviews,favorites,watchlist,profile}.ts` are the TanStack Query wrappers (favorites/watchlist toggles are optimistic). The SQL schema + RLS + signup trigger live in `supabase/schema.sql` (run once in the dashboard; see `supabase/README.md`). **Only the anon key is bundled** (via `app.config.ts` `extra`, same pattern as TMDB) — RLS is the real protection; the `service_role` key must never enter the app. Row types in `types.ts` are `type` aliases, not `interface`s, and each table carries `Relationships: []`, because postgrest-js's `GenericSchema` constraint otherwise collapses the whole `Database` to `never`.
  - **Auth is session state, not React-owned**: `store/auth-store.ts` (Zustand) holds the session; `hooks/use-auth-init.ts` (called once in the root layout) is the *only* thing that calls `setSession`, driven by `supabase.auth.onAuthStateChange`. Browsing is guest-allowed; `hooks/use-require-auth.ts` gates write actions by pushing the `/auth` modal. **Never subscribe a navigator layout (e.g. `(tabs)/_layout.tsx`) to the auth store** — read it imperatively with `useAuthStore.getState()` in handlers. A subscription there re-renders the tab tree on every auth event and can spin React Navigation's per-screen option effects into a "Maximum update depth exceeded" loop.
- **Local-only persistence (`store/`)**: `theme-store` (preference, synced into NativeWind's `colorScheme.set`) and `search-store` (recent searches) remain Zustand + AsyncStorage — pure device UX, never synced. (Diary/favorites/watchlist used to be local Zustand stores; they now live in Supabase.)
- **Theming**: NativeWind (`darkMode: 'class'`) drives `className`-based styling everywhere. `hooks/use-app-theme.ts` is the bridge for the few places that can't use `className` (icon `color` props, the navigation `ThemeProvider`, status bar style, gradients, blur tint) — it resolves NativeWind's `useColorScheme()` against the hex token table in `constants/theme.ts` (`Colors.light` / `Colors.dark`, kept in sync with `tailwind.config.js`). To add a new semantic color, add it in both places.
- **Component layers**: `components/common/` is app-agnostic UI (Button, Input, Chip, Skeleton, Modal, BottomSheet wrapper, ThemeToggle, ErrorBoundary — barrel in `index.ts`). `components/movie/` and `components/review/` are domain components (MovieCard, MovieCarousel, MovieGridScreen for paginated grids, SearchFilterSheet, StarRating, ReviewCard). `MovieGridScreen` is shared by both `browse/[category]` and `browse/genre/[id]` to avoid duplicating the grid+pagination+header chrome.
- **Path aliasing**: `@/*` maps to the project root (`tsconfig.json`).
- **New Architecture**: `newArchEnabled: true` — runs on Fabric/TurboModules. Reanimated 4 uses the split `react-native-worklets` package; the Babel plugin is `react-native-worklets/plugin`, not the old `react-native-reanimated/plugin`.
- **`babel-preset-expo` is an explicit top-level devDependency** — without it, npm's dedupe nests it under `node_modules/expo/node_modules/`, which our custom `babel.config.js` (needed for NativeWind + worklets) can't resolve from the project root.
